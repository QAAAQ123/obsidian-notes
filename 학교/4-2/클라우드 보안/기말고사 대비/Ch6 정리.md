### 📌 1. 접근 통제(Access Control) 3요소 & IAAA 모델

기본 용어 정의 문제입니다. 헷갈리지 않게 확실히 외우세요.

(1) 접근 통제 3요소 1

- **주체 (Subject):** 접근을 요청하는 행위자 (User, Process).
    
- **객체 (Object):** 보호 대상이 되는 리소스 (File, DB, Memory).
    
- **접근 권한 (Access Right):** 주체가 객체에 대해 수행할 수 있는 행위 (Read, Write, Execute).
    

(2) 접근 통제 단계 (IAAA) 2

순서대로 기억하는 것이 중요합니다.

1. **식별 (Identification):** "너 누구니?" (ID 주장). 예: ID 입력.
    
2. **인증 (Authentication):** "증명해봐." (신원 검증). 예: PW, 생체인식, MFA.
    
3. **인가 (Authorization):** "권한 있어?" (허용 범위 결정). 예: 파일 읽기 권한 확인.
    
4. **감사 (Auditing):** "뭐 했어?" (기록). 예: 로그 기록.
    
5. **책임 (Accounting):** "네 책임이야." (추적성, 과금).

### 📌 2. 접근 통제 보안 원칙 (⭐️ 필수 암기)

서술형이나 객관식 "다음 중 아닌 것은?" 문제로 자주 나옵니다.

1. **직무 분리 (SoD - Separation of Duties):** 한 사람이 모든 중요한 권한을 갖지 못하게 함 (사기 방지)3.
    
2. **최소 권한의 원칙 (PoLP - Principle of Least Privilege):** 업무 수행에 **딱 필요한 만큼만** 권한 부여4.
    
3. **알 필요의 원칙 (Need to Know):** 업무상 알아야 하는 정보만 접근 허용5.
    
4. **Fail-Safe Default:** 기본적으로 **모두 차단(Deny)**하고, 허용된 것만 연다6.
    
5. **완전한 중재 (Complete Mediation):** **모든** 접근 시도마다 권한을 검사해야 함 (캐싱된 권한 사용 주의)7.

### 📌 3. 접근 통제 구현 메커니즘 (구조 비교)

행(Row)과 열(Column)의 차이를 기억하세요.

|**구분**|**설명**|**특징**|
|---|---|---|
|**ACM (Access Control Matrix)**|주체(행)와 객체(열)의 모든 권한을 행렬로 표현|관리가 복잡하고 비효율적임 8|
|**ACL (Access Control List)**|**객체(Object) 기준**으로 접근 가능한 주체 목록 (열 중심)|"이 파일에 누가 접근 가능해?" 확인에 유리 9|
|**Capability List**|**주체(Subject) 기준**으로 접근 가능한 객체 목록 (행 중심)|"이 유저가 무슨 파일 열 수 있어?" 확인에 유리 (티켓 개념) 10|

### 📌 4. 접근 통제 모델 (DAC vs MAC vs RBAC vs ABAC)

가장 중요한 파트입니다. 각 모델의 **결정 주체**와 **특징**을 비교하세요.

|**모델**|**결정 주체**|**특징 및 장단점**|
|---|---|---|
|**DAC (임의적)**|**데이터 소유자 (Owner)**|유연하지만, 소유자 맘대로라 보안 위험이 있음 (Trojan 취약) 11|
|**MAC (강제적)**|**시스템 (중앙 관리자)**|보안 등급(Label) 기반. 가장 안전하지만 융통성 없고 복잡함 12|
|**RBAC (역할 기반)**|**조직 내 역할 (Role)**|인사 이동 시 편리함. 그룹핑하여 관리 효율성 증대 (가장 대중적) 13|
|**ABAC (속성 기반)**|**속성 + 환경 (Context)**|"누가, 언제, 어디서" 등 조건(If-then) 기반. 매우 세밀하지만 구현 복잡 14|
|**PBAC (정책 기반)**|**정책 (Policy)**|룰과 정책을 중앙에서 관리. 유연하고 동적임 15|

### 📌 5. 클라우드 IAM (AWS 예시 중심)

클라우드 보안의 핵심입니다. 구성 요소와 정책 평가 로직을 이해하세요.

(1) IAM 구성 요소 16161616

- **User:** 실제 사용자 또는 애플리케이션.
    
- **Group:** 사용자의 집합 (권한 관리 용이).
    
- **Role:** **일시적인** 권한. (사람이 쓰거나 AWS 서비스가 다른 서비스 접근할 때 사용). 자격증명이 영구적이지 않음17.
    
- **Policy:** 권한을 정의한 JSON 문서 (Allow/Deny).
    

(2) 정책 평가 로직 (순서 중요!) 18

1. **명시적 차단 (Explicit Deny):** `Deny`가 하나라도 있으면 무조건 차단.
    
2. **명시적 허용 (Explicit Allow):** `Deny`가 없고 `Allow`가 있으면 허용.
    
3. **묵시적 차단 (Implicit Deny):** 아무 언급이 없으면 기본적으로 차단.
    

#### **(3) 인증 확장 기술**

- **SSO (Single Sign-On):** 한 번 로그인으로 여러 서비스 이용19.
    
- **Federation (연합 ID):** 구글/페북 계정으로 로그인 (신뢰 관계 기반)20.

### 📌 6. IAM 보안 위험 및 대응 (서술형 대비)

- **위험:** 과도한 권한(Excessive Privileges), 약한 암호, 미사용 계정 방치, 하드코딩된 키(Credential Exposure)21212121212121212121212121212121.
    
- **대응:** MFA 적용, 주기적 권한 검토(Access Review), IaC 스캔, **최소 권한 원칙 준수**.

### 📌 7. 컨테이너 & 쿠버네티스 보안 정책 (Ch 6 후반부 핵심)

클라우드 네이티브 환경에서의 접근 통제입니다.

(1) 3가지 보안 정책 레벨 22222222

- **Network Policy:** Pod 간 통신 제어 (L3/L4). CNI 플러그인(Cilium, Calico)으로 구현.
    
- **System Policy:** 컨테이너 내부 프로세스/파일 접근 제어. 런타임 보안 도구(Falco, Tetragon) 사용.
    
- **Cluster Policy:** 클러스터 전체 리소스/설정 제어. Admission Controller(Kyverno, OPA) 사용.
    

(2) 컨테이너 격리 기술 (리눅스 커널) 23

- **Namespace:** 리소스(PID, Network 등)를 **논리적으로 격리**하여 안 보이게 함.
    
- **Cgroups:** 리소스(CPU, Memory) 사용량을 **제한**함.
    

(3) 컨테이너 런타임 보안 도구 비교 24

- **Falco:** 위협 **탐지(Detection)** 중심. (차단 기능은 약함).
    
- **KubeArmor/Tetragon:** **강제(Enforcement)** 및 차단 가능. (eBPF/LSM 기반).
