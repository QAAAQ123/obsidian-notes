- 하나의 공시 타입에 여러개의 AI 분석이 들어감(재무따로 위험따로 이런식으로)
- 3개의 테이블: 회사 정보/공시 정보/공시item별 분석 정보
- 모든 FK는 ID로 할 것

## 회사 정보(Company)
1. id(PK): int
2. ticker: string,unquie
3. name: string
4. cik: string,unquie
5. created_at: DateTime
6. updated_at: DateTime

## 공시 메타데이터(Filing)
1. id(PK): int
2. accession_number: string,unquie
3. filing_type: string
4. filing_date: Date
5. primary_document: string
6. created_at: DateTime
7. updated_at: DateTime
- **FK: company_id**

## 게시글(Post)
1. id: int
2. title: String
3. part: small-int
4. item: Varchar
5. content: text(무제한 문자열)
6. status: Enum(작성중,게시됨,실패)
7. posted_at: DateTime
8. updated_at: DateTime
9. language: Enum
- **FK: filing_id**

`10-Q는 Part 1,2이고 item이 파트별로 같다(Part 1/1,2,3,4와 Part 2/1,2,3,4)`
`10-K는 Part 1,2,3,4이고 item이 끝까지 이어진다(Part 1/1,2,3,4 Part 2/5,6,7...)`
## 관계
1. Company:Filing = 1:N
2. Filing:Post = 1:N

## 방향성
Comany<->Filing
Filing<->Post
- 삭제시 실수 방지와 개발 편의성을 위해서 양방향 관계 설정
- 어떤 회사에 대한 post인지 찾으려면 DB를 3번 조회해야 한다-> 이것 방지하기 위해서
