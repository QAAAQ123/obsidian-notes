`createGuestJwt(String guestId,String role,Log expiredMs)`
- guestId: UUID
- role: ROLE_GUEST

	- Refresh Token: 14일

### 3. Redis에 저장할 데이터
1. 프로필 (Hash): `GUEST_PROFILE:{UUID}` -> { nickname, persona } (TTL: 14일)
2. 채팅 내역 (List): `GUEST_CHAT:{UUID}` -> [msg1, msg2...] (TTL: 14일)
3. 일일 채팅 카운트 (String): `GUEST_COUNT:{UUID}` -> 3 (TTL: 14일)


### 4. 게스트 신규 가입 흐름
#### 프론트엔드
1. `닉네임`과 `페르소나`를 입력한다
2. **닉네임 중복,정책 위반 검사를 한다**-아직 추가 안함
3. `가입 완료`버튼을 누르면 HTTP Body에 `닉네임`과 `페르소나`를 담아서 Post /auth/guest/login으로 요청을 보낸다
4. `{accessToken,role}`을 http body로 받아 Access Token은 Local Storage에 저장하고 role은 상태에 저장해둔다
5. refresh token은 HttpOnly Cookie로 자동 저장된다.
#### 백엔드
1. Post: /auth/guest/login
2. 게스트 식별자인 UUID를 만든다
3. Redis `GUEST_PROFILE:{UUID}`에 정보 저장.
4. `createJwt`로 Access/Refresh Token을 만들어 프론트엔드로 보낸다



### 5. 기존 게스트 로그인 흐름
#### 프론트엔드
1. Local Storage에 Access Token을 꺼내 만료 여부를 확인한다.
2. 만료되었다면 `Post /auth/refresh`로 재발급 요청을 한다.
3. 만료되지 않았다면 그대로 유지하여 서비스를 지속한다

#### 백엔드
1. Post: /auth/refresh
2. HttpOnly Cookie로 받아온 Refresh Token안에 UUID,role,'JWT expired at'을 추출-`try-catch`문으로 토큰만료Exception(401)을 잡아내야 한다.
	1. role이 ROLE_GUEST이면 redis안에 해당하는 UUID가 있는지 확인한다.
		1. UUID가 없으면 `게스트 계정 사용기간이 만료되었습니다. 로그인 화면으로 이동합니다.`류의 HTTP 상태코드 401 응답해야한다.
	2. Refresh Token Rotation 방식을 사용했기 때문에 `createJwt`로 Access,Refresh Token을 만들어 프론트엔드로 보낸다
	3. Redis의 GUEST_PROFILE,GUEST_CHAT,GUEST_COUNT의 TTL을 연장해줘야 한다.

#### 백엔드 추가
- CustomAuthenticationEntryPoint에 GUEST_NOT_EXIST 추가해야함
	- CustomAuthenticationEntryPoint는 Access Token을 검사하여 알맞는 에러를 프론트로 보내주는 역할
- /auth/refresh에서는 Refresh token에 대한 검사이고 CustomAuthenticationEntryPoint에서는 access token에 대한 검사이니까 '게스트 UUID없음'에 대한 검사는 2개 모두 해줘야 한다.


### 6. 게스트 채팅 저장
- 첫번째 채팅을 시작하면 `GUEST_CHAT:{UUID}`과 `GUEST_COUNT:{UUID}`생성
- 백엔드로 채팅이 들어오면 `GUEST_CHAT:{UUID}`에 Lpush
	- `GUEST_CHAT:{UUID}`에 LPush할 때 `GUEST_COUNT:{UUID}`는 INCR
- `GUEST_COUNT`의 값도 프론트로 보내줘야함

	1. Local Stoarge에 있는 ROLE_GUEST에 토큰은 건드리지 않았으므로, 자연스럽게 서비스 지속 가능

## 백엔드 
- ~={red}**헤더의 토큰 유무로 게스트 가입한 사용자와 처음 가입하는 사용자를 구별**=~
- /auth/user/login과 /auth/user/signup을 따로 만들어서 이미 가입한 유저와 아닌 유저를 구별해야함
- ROLE_PENDING: accessToken(만료시간 20분)만 발급하고 refresh token은 발급하지 않는다.
	- email,식별자,role,name,espiresMs
- /auth/user/signup: Spring Security에서 토큰을 검증하고, 토큰의 Role이 정확히 `ROLE_PENDING`인지 확인한다. 아니면 403 Forbidden 에러 처리.
### 게스트로 가입한 사용자일 경우
`Post: /auth/user/login` 
`Header: Authorization: Bearer {accessToken}-ROLE_GUEST`  
`Body: {code,provider}`
1. Spring Security Filter Chain에서 Access Token을 검증하고 SecurityContext에 저장된 UUID(Principal)를 꺼냄 
2. provider에 맞는 인증서버로 code를 보내서 Access/Refresh Token을 받아옴
3. 인증서버에서 사용자 정보(email,실명)을 가져옴
4. Redis에 실제 UUID가 존재하는지 확인
5. ROLE_PENDING 토큰을 생성함**★★★(ROLE_PENDING을 생성할 때 email,uuid,role,name,provider,expiresMS를 담아야함)★★★**
6. Http Body에 {accessToken,ROLE_PENDING,nickname,persona,lang}을 담아서 프론트로 보냄

`Post: /api/onboarding/complete`
`Header: Authorization: Bearer {accessToken}-ROLE_PENDING`
`Body: {nickname,birthdate,mbti,sasng 등 입력한 정보}`
1. Spring Security에서 토큰을 검증하고, 토큰 안에 들어있던 `email`과 `guestUuid`를 꺼냄
2. Redis에서 `guestUuid`를 이용해 채팅 기록 등 기존 데이터를 가져옴
3. Body로 받은 정보(mbti 등)와 토큰에서 꺼낸 `email`, Redis에서 꺼낸 게스트 데이터를 모두 합쳐서 DB에 최종 회원(Member)으로 저장 
4. Redis에 있던 해당 게스트 데이터는 삭제
5. `ROLE_USER` 토큰 생성 후 프론트로 응답하여 로그인 완료


### 게스트로 가입없이 바로 구글/카카오로 가입하는 경우
`Post: /auth/user/login` 
`Body: {code,provider}`
1. provider에 맞는 인증서버로 code를 보내서 Access/Refresh Token을 받아온다
2. Access Token을 보내서 사용자 정보를 가져온다.
3. 사용자 정보가 DB에 이미 있는지 검사한다.
4. 없기 때문에 ROLE_PENDING인 Access만 발급한다.(이때 PENDING 토큰에 email,"new_user",role,name,provider,expiresMs를 넣는다)
5. Body에 {role,accessToken}을 담아서 로그인을 완료한다.Token의 Role은 ROLE_PENDING이다.

`Post: /api/onboarding/complete`
`Header: Authorization: Bearer {accessToken}-ROLE_PENDING`
`Body: {nickname,birthdate,mbti,sasng 등 입력한 정보}`
1. 토큰을 검증하고, 토큰 안에 들어있던 'email'을 꺼냄
2. Body로 받은 정보와 email을 종합하여 DB에 신규 회원으로 저장한다.
3. ROLE_USER로 토큰 생성후 프론트포 응답하여 회원가입 완료

### 이미 가입한 유저가 로그인 할 경우
`Post: /auth/user/login`
`Body: {code,provider}`
1. provider에 맞는 인증서버로 code를 보내서 Access/Refresh Token을 받아온다
2. Access Token을 보내서 사용자 정보를 가져온다.
3. 사용자 정보가 DB에 이미 있는지 검사한다.
4. 있기 때문에 ROLE_USER인 Access/Refresh를 발급한다.
5. Body에 {role,accessToken}을 담아서 로그인을 완료한다.Token의 Role은 ROLE_USER이다.

### 공통 사항
- **요청(Request, 줄 때):** "나 누구야"라고 신분증을 제시하는 것이므로 **Header(`Authorization`)**에 담는 것이 표준입니다.
- **응답(Response, 받을 때):** "자, 여기 네가 요청한 데이터(토큰, 정보)야"라고 주는 것이므로 **Body(JSON)**에 담는 것이 표준입니다. (OAuth 2.0 표준 규격도 토큰 발급 응답은 JSON Body로 줍니다.)

### OAuth SDK 리다이렉트 방식 흐름
1. 프론트가 JS함수로 로그인 요청을 한다.
2. JS 라이브러리가 알아서 알맞은 형태의 로그인을 한다.
3. 끝나면 인증서버의 설정에 등록했던 Redirect URI에 `callback?code=xxx`형태로 리다이렉트 URI를 받는다.(우리는 프론트가 callback을 받음)
4. 이 code와 게스트 Access token을 Post방식으로 백엔드에 요청을 보낸다.
5. 백엔드는 code를 가지고 인증서버에 token요청,유저정보 요청을 순서대로 보내어 원하는 유저 정보를 획득한다.
6. JWT 토큰을 발급하여 프론트로 보낸다.


### 구글OAuth2.0 Code,Token 요청
[링크](https://developers.google.com/identity/protocols/oauth2/web-server?hl=ko#httprest_6)
#### code 요청
- 요청 방식:GET
- 엔드포인트: `https://accounts.google.com/o/oauth2/v2/auth`
- 쿼리 파라미터: 필수만 작성함
	1. `?client_id={google_client_id}`
	2. `&redirect_uri={uri}`
	3. `&response_type=code`: 불변
	4. `&scope=`
 - 결과: `?code=xxxx`
#### access,refresh token 요청
- 요청 방식: POST
- 엔드포인트: `https://oauth2.googleapis.com/token`
- Http body 포함 값: 순서 상관없음
	1. `client_id={client_id}`
	2. `&client_secret={client_secret}`: 선택사항
	3. `&code={code}`
	4. `&redirect_uri={uri}`
	5. `&grant_type=authorization_code`: 불변
- 응답
```json
{
  "access_token": "1/fFAGRNJru1FTz70BzhT3Zg",
  "expires_in": 3920,
  "token_type": "Bearer",
  "scope": "https://www.googleapis.com/auth/drive.metadata.readonly https://www.googleapis.com/auth/calendar.readonly",
  "refresh_token": "1//xEoDL4iW3cxlI7yDbSRFYNG01kVKM2C-259HOF2aQbI"
}
```
