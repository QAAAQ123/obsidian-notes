## 게스트
### 1. 사용자로부터 받을 데이터
1. 닉네임
2. 페르소나 선택
### 2. 토큰 만들때 들어가는 인자
`createJwt(String email, String role, Long expiredMs)`=> `email`을 `identifier`로 바꿀 예정
- identifier: 게스트 로그인 할 때 백엔드에서 만든 UUID(식별자)값
- role: ROLE_GUEST
- expiredMs
	- Access Token: 1시간
	- Refresh Token: 14일

### 3. Redis에 저장할 데이터
1. 프로필 (Hash): `GUEST_PROFILE:{UUID}` -> { nickname, persona } (TTL: 14일)
2. 채팅 내역 (List): `GUEST_CHAT:{UUID}` -> [msg1, msg2...] (TTL: 14일)
3. 일일 채팅 카운트 (String): `GUEST_COUNT:{UUID}` -> 3 (TTL: 매일 자정까지)


### 4. 게스트 신규 가입 흐름
#### 프론트엔드
1. `닉네임`과 `페르소나`를 입력한다
2. `가입 완료`버튼을 누르면 HTTP Body에 `닉네임`과 `페르소나`를 담아서 Post /auth/guest/login으로 요청을 보낸다
3. `{accessToken,role}`을 http body로 받아 Access Token은 Local Storage에 저장하고 role은 상태에 저장해둔다
4. refresh token은 HttpOnly Cookie로 자동 저장된다.
#### 백엔드
1. Post: /auth/guest/login
2. 게스트 식별자인 UUID를 만든다
3. Redis `GUEST_PROFILE:{UUID}`에 정보 저장.
4. `createJwt`로 Access/Refresh Token을 만들어 프론트엔드로 보낸다



### 5. 기존 게스트 로그인 흐름
#### 프론트엔드
1. Local Storage에 Access Token을 꺼내 만료 여부를 확인한다.
2. 만료되었다면 `Post /auth/refresh`로 재발급 요청을 한다.
3. 만료되지 않았다면 그대로 유지하여 서비스를 지속한다

#### 백엔드
1. Post: /auth/refresh
2. HttpOnly Cookie로 받아온 Refresh Token안에 UUID,role,'JWT expired at'을 추출-`try-catch`문으로 토큰만료Exception(401)을 잡아내야 한다.
	1. role이 ROLE_GUEST이면 redis안에 해당하는 UUID가 있는지 확인한다.
		1. UUID가 없으면 `게스트 계정 사용기간이 만료되었습니다. 로그인 화면으로 이동합니다.`류의 HTTP 상태코드 401 응답해야한다.
	2. Refresh Token Rotation 방식을 사용했기 때문에 `createJwt`로 Access,Refresh Token을 만들어 프론트엔드로 보낸다

#### 백엔드 추가
- CustomAuthenticationEntryPoint에 GUEST_NOT_EXIST 추가해야함
	- CustomAuthenticationEntryPoint는 Access Token을 검사하여 알맞는 에러를 프론트로 보내주는 역할
- /auth/refresh에서는 Refresh token에 대한 검사이고 CustomAuthenticationEntryPoint에서는 access token에 대한 검사이니까 '게스트 UUID없음'에 대한 검사는 2개 모두 해줘야 한다.


### 6. 게스트 채팅 저장
- 첫번째 채팅을 시작하면 `GUEST_CHAT:{UUID}`과 `GUEST_COUNT:{UUID}`생성
- 백엔드로 채팅이 들어오면 `GUEST_CHAT:{UUID}`에 Lpush
	- `GUEST_CHAT:{UUID}`에 LPush할 때 `GUEST_COUNT:{UUID}`는 INCR
- `GUEST_COUNT`의 값도 프론트로 보내줘야함

## 카카오/구글 회원가입 흐름
- 구글 Oauth 방식: `Google Identity Services(GSI) SDK`
### 프론트엔드
1. SDK 방식으로 google/kakao 인증 서버로 `code` 요청 
2. 프론트에서 `callback?code=xxxx`형식으로 `code`를 받음(kakao와 google의 `로그인 리다이렉트 URI`을 수정해야함)
3. `{code,provider}`를 Http Body에 담고 게스트의 access token은 Header에 담아서 `POST:/auth/user/login`으로 로그인 요청을 보냄
4. 백엔드에서 로그인을 완료한 후  {AccessToken,Role}을 Http Body에 담아서 응답함
5. 게스트 로그인시에 발급하여 Local Storage에 저장해 놓은 Access Token을 삭제
6. 회원가입 완료
## 백엔드 
`Post: /auth/user/login` 
`Header: Authorization: Bearer {accessToken}` 
`Body: {code, provider}` 
조건: 헤더에 유효한 게스트 토큰(ROLE_GUEST)이 있는 요청만 Security Filter Chain 통과 가능 
1. Spring Security Filter Chain에서 Access Token을 검증하고 SecurityContext에 저장된 UUID(Principal)를 꺼냄 
2. 꺼낸 UUID가 Redis(GUEST_PROFILE:{UUID})에 실제 저장되어 있는지 확인 
3. 조건이 맞지 않으면(토큰 만료, Redis 키 소멸 등) 401 Unauthorized 또는 가입 실패 예외 발생 
4. 프론트에서 Body로 받아온 provider에 맞는 인증 서버로 code를 보내 인증서버-백엔드간의 AccessToken과 RefreshToken을 받아옴 
5. AccessToken으로 가입할 유저의 정보를 인증 서버로부터 가져옴 
6. Redis에 있던 닉네임, 페르소나, 채팅기록을 꺼내서 인증 서버에서 받은 email과 연동시켜 DB에 저장 
7. 인증 서버에서 받아온 실명, 이메일, provider를 DB에 저장 
8. Redis에서 회원가입한 UUID의 모든 데이터(프로필, 채팅, 카운트) 삭제 
9. email과 ROLE_USER로 프론트-백간 통신에 사용할 Access Token과 Refresh Token을 생성함 
10. Access Token과 Role을 프론트로 전송

### 공통 사항
- **요청(Request, 줄 때):** "나 누구야"라고 신분증을 제시하는 것이므로 **Header(`Authorization`)**에 담는 것이 표준입니다.
- **응답(Response, 받을 때):** "자, 여기 네가 요청한 데이터(토큰, 정보)야"라고 주는 것이므로 **Body(JSON)**에 담는 것이 표준입니다. (OAuth 2.0 표준 규격도 토큰 발급 응답은 JSON Body로 줍니다.)
