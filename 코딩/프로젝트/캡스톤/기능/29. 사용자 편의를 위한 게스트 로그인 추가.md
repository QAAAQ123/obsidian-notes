## 확인완료사항
- 프론트에서 Bearer를 붙혀줌
- redis에 refresh token을 key-value(String)으로 저장함(key: "RT: `email`")
- refresh token을 전달할 때 HTTP 쿠키로 전달함
- access token은 URI Query나 HTTP Body로 전달함

## 게스트 로그인 추가
- filter chain이 잡아서 자동 발급 못해줌
- 하지만 filter chain에 addfilter로 추가해주면 안된다.
	1. JSON 데이터 처리 복잡성-ObjectMapper를 이용해 수동 JSON 처리 로직 만들어야함
	2. spring security filter chain의 목적에 맞지 않다.-[[30. Spring security filter chain의 목적과 addFilter 사용법]]
	3. 예외 처리의 어려움-AOP나 GlobalExecptionHandler가 동작하지 않는다
- 따라서, 컨트롤러로 만들어야 한다
	1. POST: `/auth/login/guest`
		- 유저가 입력한 정보
	2. jwt 토큰 2개 생성
	3. redis에 유저가 입력한 정보,refreshToken,role 저장해 둬야함
- Redis에 `RT: UUID`로 저장해야함
- 토큰을 떼어와서 Redis UUID와 토큰의 UUID가 일치하는지 항상 검사해야한다.
- `RT: UUID`가 만료되었는지 확인해야함

## 카카오,구글 회원가입
- redis에 들어가 있는 사용자 정보를 꺼내와서 회원가입해야한다.
	- 기존의 memberDTO에서 가져오는 로직을 수정해야한다.
	- 회원가입 완료 버튼을 누르기 전까지 유저는 ROLE_GUSET여야 한다.
- 프론트는 화이트리스트 방식으로 권한 인증을 구현해야 한다.
- 토큰을 떼어내고 UUID를 확인하여 Reids에 있는지 확인해야한다.
	- 어떤 게스트가 회원가입 하려는지 모르기 때문에
- 구글 filter chain 자동 로그인 코드와 kakao 수동 로그인 코드에 프론트에서 보낸 UUID가 Redis에 있는지 확인하는 코드를 작성해야 한다.
- 회원가입 완료(ROLE_USER)후 Redis 게스트 UUID key-value 삭제

## 구글 Auth회원가입 방법 변경
- 구글은 filter chain이 자동으로 리다이렉트 해주는 방법을 사용하고 있었다. 
- 그런데,이 방법은 어떤 게스트 계정이 회원가입을 하는지 확인해야 하는 과정을 거쳐야 한다.
- 프론트에서 날아온 access token을 백엔드에서 받아서 redis에 있는 게스트 UUID중 하나와 일치하는지를 확인해야 한다.
- 프론트 -> 구글 oauth 서버-> 프론트에서 code를 받아옴 -> 프론트에서 access token과 code를 동시에 전달-> 백엔드에서 받아서 access token을 디코딩하고 redis에 있는 uuid와 비교-> uuid가 있으면 redis의 정보를 서비스 로직에 저장 -> 구글 서버로 code를 보내 토큰2개 받음 -> 접근 토큰을 다시 구글 서버로 보내서 사용자 정보 받음 -> 구글에서 받은 사용자 정보와 redis에 저장되었던 정보를 MongoDB에 넣음 -> 토큰 생성 -> 전달
