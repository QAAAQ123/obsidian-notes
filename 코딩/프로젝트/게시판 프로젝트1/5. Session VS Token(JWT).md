## Session 인증 방식
- Session 방식은 사용자가 로그인 하면 입장권 발급해서 주고, 입장권 정보를 DB에 저장해 놓는다. 입장 요청이 있을 때마다 DB에서 검사하여 입장을 허가해준다
- ==Stateful 한 방식이다==

## Token 인증 방식
- Token 방식은 사용자가 로그인 하면, 로그인 한 데이터와 각종 정보를 가지고 있는 Token을 생성한다. 입장 요청이 있으면 Token만 확인하여 입장을 허가해 준다.
- JWT(JSON Web Token) 방식은 Token 방식중 데이터 형태가 JSON인 Token 인증 방식이다
- ==Stateless 한 방식이다==

## Session 인증 방식과 Token 인증 방식의 차이점
- Token 방식
	- 토큰(특히 JWT)은 사용자 정보(예: 사용자 ID, 권한 등)와 토큰 발행 시간, 만료 시간 등의 데이터를 **자체적으로(Self-contained)** 포함하고 있습니다.
	- 토큰이 서버로 전송되면, 서버는 토큰의 유효성(서명 검증, 만료 여부 등)만 확인하고 토큰 내부의 정보를 바로 사용하여 사용자를 식별하고 권한을 부여합니다.
	- 이러한 특성 때문에 서버는 클라이언트의 상태를 저장할 필요가 없어 **Stateless(무상태성)**하며, 서버 확장이 용이하다는 장점이 있습니다. 여러 서버가 있더라도 토큰만 유효하면 어떤 서버든 처리가 가능합니다.
- Session 방식
	- 사용자가 로그인하면 서버는 고유한 **세션 ID**를 생성하고, 이 세션 ID와 관련된 사용자 정보를 서버의 메모리나 데이터베이스, 또는 별도의 세션 저장소(Redis 등)에 저장합니다.
	- 서버는 이 세션 ID를 사용자에게 쿠키 형태로 전송하고, 사용자는 이후 모든 요청에 이 세션 ID를 담은 쿠키를 함께 보냅니다.
	- 서버는 요청 시마다 쿠키에서 세션 ID를 추출하여, 서버에 저장된 세션 정보를 조회하여 사용자를 식별하고 인가를 확인합니다.
	- 이 방식은 서버가 사용자 로그인 상태를 유지해야 하므로 **Stateful(상태 유지)**하며, 서버 확장에 어려움이 있을 수 있습니다. (모든 서버가 동일한 세션 정보를 공유해야 함)
- 결론
	- **토큰:** 클라이언트 측에 인증 정보가 자체적으로 담겨있고, 서버는 검증만 함 (Stateless)
	- **세션:** 서버 측에 인증 정보가 저장되고, 클라이언트는 세션 ID만 가짐 (Stateful)
