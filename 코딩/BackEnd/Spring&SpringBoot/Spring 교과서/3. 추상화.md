### 배울 것
- 추상화를 사용하는 방법
- 프로젝트에서 구현을 분리하는 데 추상화를 사용할 때가 많아 매우 중요

## 구현 분리
- 자바에서는 인터페이스로 구현을 분리한다. 인터페이스를 통해 구현 간 계약을 정의한다고도 말한다. 
- 쉽게 말해서 구현할 기능을 인터페이스에 넣어 놓고, 구현은 클래스에서 한 다음 그 객체를 빈으로 등록하는 것을 말한다. 이렇게 **~={red}작성하면 기능에만 집중 할 수 있고, 기능의 세부 사항이 바뀌더라도 개발자는 신경 쓸 필요가 없다. =~**

## 예시
문자 서비스를 예시로 구현 분리를 설명함
1. @Service CommnetService가 CommentRepository Interface와 CommentNotificationProxy Interface를 사용함
2. **CommentRepository는 @Repository DBCommentRepository에서 구현한다**
3. **CommentNotificationProxy는 @Component EmailCommnetNotificationProxy에서 구현한다.** 
4. 최종적으로, 콘텍스트에 있는 객체 인스턴스는 CommnetService,DBCommentRepository,EmailCommnetNotificationProxy 이다.
5. 두개의 Interface는 객체가 될 수 없으므로 컨텍스트에 올라가지 않는다.

## CommnetService에서 메소드를 구현하지 않은 CommentRepository(interface)를 사용하는데 구현한 메소드를 사용할 수 있는 이유
- 예시1번에서 CommentSerive는 CommentRepository **~={red}인터페이스=~**를 사용한다.
- 인터페이스가 컨텍스트에 올라가 있지 않고, 메소드를 구현하지 않았는데에도 메소드의 기능을 사용할 수 있는 이유는
- ★★★**~={red}스프링이 인터페이스의 메소드를 구현한 빈을 검색하는 방법을 알고 있기 때문이다.=~** ★★★
- DBCommnetRepositry와 EmailCommnetNotificationProxy를 알아서 찾는다

## 구현체 중 어떤 것을 주입할지 결정
앞에서 배운 @Primary와 @Qualifier를 통해 정하면 된다.
- CommentNotificationProxy Interface의 구현체가 2개 이상이고 모두 같은 sendComment()메소드를 구현한다고 가정한다.
	- 위 상황에서 결정은 '2. 빈 작성'에서 배운 대로 @Primary나 @Qulifier로 결정한다.

## 스테레오 타입 어노테이션
다양한 스테레오 타입 어노테이션을 통해 @Component보다 명시적으로 컴포넌트의 목적을 정의 할 수 있다.
1. `@Component`: 가장 일반적인 스프링 빈으로 등록될 클래스에 사용합니다. 
2. `@Service`: 비즈니스 로직을 처리하는 서비스 계층의 클래스에 사용합니다. `@Component`의 특별한 형태입니다.
3. `@Repository`: 데이터베이스와 통신하는 데이터 접근 계층(DAO)의 클래스에 사용합니다. `@Component`의 특별한 형태이며, 데이터베이스 관련 예외를 스프링의 예외로 자동 변환하는 기능을 제공합니다.
4. `@Controller`: 웹 요청을 처리하는 컨트롤러 계층의 클래스에 사용합니다. `@RequestMapping`과 같은 어노테이션과 함께 사용됩니다.
5. `@RestController`: RESTful 웹 서비스를 만드는 컨트롤러에 사용합니다. `@Controller`와 `@ResponseBody`의 기능을 결합한 어노테이션으로, 메서드의 반환값을 JSON이나 XML 형태로 바로 응답합니다.
