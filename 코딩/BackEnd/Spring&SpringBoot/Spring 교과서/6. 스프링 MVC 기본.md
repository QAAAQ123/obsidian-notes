### 웹앱이란?
웹앱(Web Application)은 웹 브라우저를 통해 접속하고 사용하는 애플리케이션

**웹사이트와 달리 사용자와 상호작용하는 복잡한 기능**을 제공하는 것이 특징입니다. 예를 들어, 온라인 쇼핑몰, 소셜 미디어, 웹 기반 이메일 서비스 등이 웹앱에 해당합니다. 별도의 설치 과정 없이 인터넷만 연결되면 어디서든 접근할 수 있다는 장점이 있습니다.프런트엔드와 백엔드의 상호작용입니다.

#### 스프링으로 웹앱을 구현하는 방식
1. 클라이언트의 요청에 응답하여 백엔드가 완성된 뷰를 제공하는 앱(서버사이드 랜더링)-소규모 앱에 적합
2. 프런드엔드-백엔드 분리 방식의 앱(클라이언트 사이드 랜더링)-대규모 앱에 적합

## 서블릿 컨테이너(Servlet container)
- 서블릿 컨테이너(웹 서버): **~={red}HTTP 요청과 응답을 자바 앱에 맞게 변환해주는 역할=~**
- 서블릿: 서블릿 컨테이너와 직접 상호작용하는 자바 객체/앱 로직의 진입점
- 서블릿 컨테이너를 사용하면 자바 앱에서 통신 계층을 구현할 필요가 없어진다.
- 서블릿 컨테이너의 구현체 중에 톰캣을 가장 많이 사용한다.

#### 동작 과정
1. 클라이언트가 특정 경로(/path)에 요청을 보내면 서블릿 컨테이너는 **해당 경로에 등록된 서블릿을 호출**한다
2. 서블릿 컨테이너는 HTTP 요청을 변환하고 해당 경로와 연관된 서블릿을 호출할 때 자바 객체로 응답한다.
3. 서블릿 컨테이너는 여러 서블릿 인스턴스를 등록한다. 각 인스턴스는 특정 경로와 대응하는 서블릿 자바 객체이다.


## 스프링 부트 장점
1. 간소화된 프로젝트 생성: 프로젝트 초기화 서비스를 이용하여 비어있는 뼈대 앱을 쉽게 얻을 수 있다.
2. 의존성 스타터: 개발자가 서로 호환되는 의존성 버전을 확인하지 않아도 자동으로 호환 버전을 맞춰주는 것
   의존성 스타터가 없다면 톰캣,스프링AOP,스프링 컨텍스트,스프링 시큐리티 등 추가하려는 모든 의존성의 버전을 개발자가 수동으로 맞춰야 한다. 
3. 구성보다 관례(Convention-over-configuration)원칙을 적용(Autoconfiguraiton 제공)


## HTTP 요청 및 반환 흐름(MVC 서버사이드 랜더링)
![[스프링 부트 HTTP 흐름.jpg]]

흐름
1. 클라이언트가 HTTP 요청
2. 디스패쳐 서블릿이 핸들러 매핑을 이용하여 해당 경로에 대한 컨트롤러 액션을 찾음
3. 디스패쳐 서블릿이 컨트롤러의 액션을 호출
4. 디스패쳐 서블릿이 뷰 리졸버를 이용하여 뷰를 찾음
5. 디스패쳐 서블릿이 뷰를 HTTP 응답으로 반환

---
상세 흐름
1. 클라이언트가 HTTP 요청
2. 톰캣이 HTTP 요청 수신
3. 톰캣은 HTTP 요청에 맞는 서블릿(디스패쳐 서블릿)을 호출
4. 디스패쳐 서블릿이 컨트롤러를 찾기 위해 핸들러 매핑에 역할을 위임한다.핸들러 매핑은 @RequestMapping이다
5. 핸들러 매핑이 찾은 컨트롤러를 디스패쳐 서블릿에서 호출하여 컨트롤러 액션을 한다.
6. 컨트롤러는 뷰를 디스패쳐 서블릿에 반환한다.이때 뷰를 가져오는 책임은 뷰 리졸버에게 있다.
7. 디스패쳐 서블릿이 랜더링된 뷰를 HTTP 응답으로 반환한다.
HTTP 요청->디스패쳐 서블릿->핸들러 매핑->디스패쳐 서블릿->컨트롤러->뷰리졸버->디스패쳐 서블릿->HTTP 응답
