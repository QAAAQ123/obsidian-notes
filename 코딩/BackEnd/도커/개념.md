## 🐳 Docker 핵심 개념 

### 1. 철학: "내 컴퓨터에선 되는데 왜 네 컴퓨터에선 안 되지?" 해결
- **Write Once, Run Anywhere:** 한 번 만들면(Build), 어디서든(Anywhere) 똑같이 돌아갑니다.
- **Immutable (불변성):** 배포할 때마다 서버를 고쳐 쓰는 게 아니라, **기존 것을 버리고 새것(컨테이너)을 통째로 갈아 끼우는 방식**입니다. 깔끔하고 찌꺼기가 남지 않습니다.
### 2. 아키텍처 (간소화됨)
복잡한 소켓 통신은 몰라도 됩니다. 딱 3가지 요소만 기억하세요
1. **클라이언트 (Client):** `docker build`, `docker run` 명령어를 치는 당신의 터미널.
2. **데몬 (Daemon/Server):** 실제 명령을 수행하는 일꾼 (백그라운드에서 실행됨).
3. **레지스트리 (Registry):** 이미지를 저장하는 창고 (예: Docker Hub). 깃허브(Github)와 같은 역할입니다.
### 3. 가장 중요한 두 가지: 이미지 vs 컨테이너
- **이미지 (Image):** **"붕어빵 틀"** (실행 파일, 라이브러리, 설정이 모두 얼려있는 상태). 읽기 전용(Read-only).
- **컨테이너 (Container):** **"붕어빵"** (이미지를 바탕으로 실제 실행되는 프로세스). 쓰기 가능하지만, 삭제하면 데이터도 사라집니다.
## 🌐 컨테이너 네트워킹 (현대적 방식)
_과거의 복잡한 브리지 설정이나 링크(--link) 개념은 삭제했습니다._
1. **격리된 네트워크:** 도커 컨테이너는 기본적으로 외부와 차단된 **'가상 사설망'** 안에 있습니다.
2. **포트 포워딩 (Port Forwarding):** 외부(내 컴퓨터/사용자)에서 접속하려면 구멍을 뚫어야 합니다.
    - `-p 8080:8080` → 내 컴퓨터의 8080으로 들어오면 컨테이너의 8080으로 연결해줘!
3. **도커 네트워크 (Docker Network) ⭐중요:**
    - 같은 네트워크 안에 있는 컨테이너끼리는 **IP 몰라도 서로 이름으로 통신** 가능합니다.
    - 예: Spring이 MySQL을 찾을 때 `localhost`가 아니라 `mysql-container`라는 이름으로 접속.

## 💾 데이터 관리 (추가됨 - 필수 개념)
- **볼륨 (Volume):** 컨테이너는 삭제하면 안의 데이터가 다 날아갑니다.
- **해결책:** DB 데이터처럼 사라지면 안 되는 파일은 **호스트(내 컴퓨터)의 폴더와 연결(마운트)**해둡니다. 이렇게 하면 컨테이너를 지웠다 다시 만들어도 데이터는 살아있습니다.

## 🛠️ 도커 워크플로우 (Modern)

### 1. Dockerfile (요리 레시피)
- 어떤 OS를 쓰고, 어떤 파일을 복사하고, 어떤 명령어를 실행할지 적어둔 명세서.
- **계층(Layer) 구조:** 도커는 빌드 과정을 한 줄 한 줄 '레이어'로 쌓습니다.
    - _장점:_ 코드 한 줄만 수정했다면, 바뀐 부분(마지막 레이어)만 다시 빌드하므로 속도가 엄청 빠릅니다.

### 2. Docker Compose (오케스트레이션 입문) ⭐
- _원본에는 없던 내용이지만 현재 프로젝트의 핵심입니다._
- `docker run` 명령어를 일일이 칠 필요 없이, `docker-compose.yml` 파일 하나에 **백엔드, AI, DB 설정을 한 번에 정의**하고 실행합니다.

### 3. 환경 변수 (Environment Variables)
- 비밀번호나 API 키 같은 민감한 정보는 이미지 안에 박제하지 않습니다.
- 컨테이너를 실행할 때 **환경 변수로 주입**합니다. (보안 필수)
