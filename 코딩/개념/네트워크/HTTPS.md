HTTPS: Hyper Text Transfer Protocol Secure
- HTTP를 사용하면 아이디와 비밀번호가 평문 그대로 전송한다
- HTTPS를 사용하면 아이디와 비밀번호를 암호화하여 전송한다
- HTTPS를 사용하면 신뢰할 수 있는 사이트인지 판별해준다

### HTTPS 사용 이유
- 내가 사이트에 보내는 정보들을 제 3자가 못보게 한다.
- 접속한 사이트가 믿을 만한 곳인지 알려준다-기관에서 검증한 사이트만 HTTPS 사용이 허가된다
### HTTP
- 프로토콜
- 서버와 클라이언트간 통신에서 데이터를 읽고 해독하는 형식중 하나이다

### 대칭키와 비대칭키(공개키)-정보들을 제 3자가 못보게 한다
- 대칭키: 1개의 key를 사용/송수신측 모두에서 같은 복호화 방식 사용하는 것(복호화 표만 지키면 된다)
	- 복호화 방식(표)를 한번은 전송해야 하는데, 이때 탈취당하면 의미가 없다-> 비대칭키 방식이 나옴
- 비대칭키(공개키): 2개의 key를 사용
	- A를 이용해 암호화한 키는 B로만 복호화가 가능하고, B를 이용해 암호화한 키는 A로만 복호화가 가능하다
	- 과정
		- 두개의 key중 하나는 개인키로 private하게 보관하고 다른 하나의 key는 공개키로 public하게 공개한다.
		- 공개키로 암호화하여 서버로 보낸다
		- 공개키를 받은 서버가 개인키를 사용하여 복호화한다

### 접속한 사이트가 믿을 만한 곳인지 알려준다
- 인증된 서버에서 클라이언트로 보내는 정보들 중 몇개는 개인키로 암호화되어 전송된다.
- 클라이언트에서 공개키로 복호화를 할 수 있으면 인증된 서버에서 보낸 정보가 맞다는 것이다
- 신뢰가능한 기관에서 공개키만 검증해주면 된다

### CA(Certificate Authority)-공인된 민간 기업
- 정품인지 신뢰할 수 있는 기업에서 인증
- 웹 브라우저에 CA 목록이 내장되어있다
- 순서
	- 서버-클라이언트간 탐색(Handshake)
		- 클라이언트가 랜덤 데이터를 생성해서 서버에 보냄
		- 서버는 데이터를 받고 랜덤 데이터와 **인증서**를 클라이언트에 보냄
	- 클라이언트는 인증서가 진짜인지 브라우저의CA를 통해 확인을 함(비대칭키 시스템)
		- CA는 해당 CA의 개인키로 암호화 되어있음
		- 클라이언트에서 공개키를 이용해 CA인증서를 복호화 가능하면 진짜임 
			- 복호화 하면 서버의 공개키가 포함되어 있다
	- 확인후 주고 받는 데이터들은 대칭키를 사용한다
		- 비대칭키는 연산량이 많기 때문에 대칭키로 주고 받는다.
		- 대칭키를 클라이언트에게 공유할때 비대칭키를 이용해 암호화 한다
			- Handshake할때 클라이언트에 있는 랜덤 데이터 2개를 조합해 임시키를 만듦
			- 임시키를 공개키로 암호화되어서 서버로 보냄
			- 양쪽에서 일련의 과정을 거쳐 동일한 대칭키가 만들어짐
