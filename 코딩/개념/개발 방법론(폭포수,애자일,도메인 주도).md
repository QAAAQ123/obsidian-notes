
## 폭포수 개발(Waterfall Development)

폭포수 개발은 **소프트웨어 개발을 순차적인 단계로 나누어 한 단계가 완료되면 다음 단계로 진행하는 전통적인 개발 방법론**입니다.

**개발 단계:**
1. **요구사항 분석** → 2. **설계** → 3. **구현** → 4. **테스트** → 5. **배포** → 6. **유지보수**

**특징:**
- **순차적 진행**: 이전 단계가 완료되어야 다음 단계 시작
- **명확한 문서화**: 각 단계마다 상세한 문서 작성
- **되돌아가기 어려움**: 한 번 진행된 단계를 수정하기 복잡

**예시로 이해하기:** 게시판을 만든다면, **요구사항을 완전히 정의 → 전체 설계 완료 → 모든 기능 구현 → 전체 테스트 → 배포**하는 순서로 진행합니다. 중간에 "댓글 기능을 추가하고 싶다"고 하면 요구사항 단계부터 다시 시작해야 합니다.

**장점:**
- 명확한 계획과 일정 관리
- 대규모 프로젝트에 적합
- 체계적인 문서화

**단점:**
- 변경 요구사항에 대응 어려움
- 늦은 피드백으로 위험 부담 증가

**간단히 말하면**, 물이 폭포처럼 위에서 아래로만 흐르듯 개발 과정도 한 방향으로만 진행하는 체계적이지만 경직된 개발 방식입니다.



## 애자일 개발(Agile Development)

애자일 개발은 **변화에 빠르게 대응하며 고객과의 협력을 통해 소프트웨어를 개발하는 방법론**입니다.

**핵심 원칙:**
- **개인과 상호작용** > 프로세스와 도구
- **작동하는 소프트웨어** > 포괄적인 문서
- **고객과의 협력** > 계약 협상
- **변화에 대응** > 계획을 따르기

**특징:**
- **짧은 반복 주기(Sprint)**: 1-4주 단위로 개발하고 배포
- **점진적 개발**: 완벽한 기능을 한 번에 만들지 않고 단계적으로 개선
- **지속적인 피드백**: 사용자 반응을 빠르게 수집하고 반영

**예시로 이해하기:** 게시판을 만든다면, 처음부터 모든 기능(글쓰기, 댓글, 검색, 관리자)을 완벽하게 설계하지 않고, **1주차에는 글쓰기만, 2주차에는 댓글 기능을 추가**하는 식으로 단계적으로 발전시킵니다.

**장점:**
- 변화하는 요구사항에 유연하게 대응
- 빠른 시장 출시 가능
- 위험 부담 감소

**간단히 말하면**, 완벽한 계획보다는 빠른 실행과 개선을 통해 더 나은 결과를 만드는 개발 방식입니다.


## 도메인 주도 설계(DDD,Domain-drivendesign)
기술보다 비즈니스를 먼저 이해하는 방법

도메인 주도 설계는 **복잡한 비즈니스 문제를 해결하기 위해 도메인(==업무 영역==)을 중심으로 소프트웨어를 설계하는 방법론**입니다.

**핵심 개념:**

- **도메인**: 해결하고자 하는 문제 영역 (예: 쇼핑몰의 주문, 결제, 배송)
- **도메인 모델**: 비즈니스 규칙과 로직을 코드로 표현한 것
- **유비쿼터스 언어**: 개발자와 도메인 전문가가 공통으로 사용하는 용어

**예시로 이해하기:** 게시판을 만든다면, 기술적 관점에서 "데이터베이스 테이블 설계"부터 시작하지 않고, **"게시글이란 무엇인가? 사용자는 어떻게 글을 작성하고 관리하는가?"**라는 비즈니스 관점에서 시작합니다.

**장점:**
- 비즈니스 요구사항이 코드에 명확히 반영됨
- 개발자와 기획자 간 소통 개선
- 복잡한 업무 로직을 체계적으로 관리

**간단히 말하면**, ~={red}기술보다 비즈니스를 먼저 이해=~하고, 그 이해를 바탕으로 소프트웨어를 설계하는 접근 방식입니다.****

## 실제 사례를 통한 개발 방법론의 이해

### 사례: "온라인 쇼핑몰" 개발 프로젝트

---

### 1. 폭포수 개발 적용 사례

**대기업 B2B 쇼핑몰 구축 (6개월 프로젝트)**

**1단계 - 요구사항 분석 (1개월)**
- 고객사와 모든 기능 명세서 작성 (상품관리, 주문, 결제, 배송 등)
- 300페이지 분량의 상세 요구사항 문서 완성
- 고객 승인 후 변경 불가 계약 체결

**2단계 - 설계 (1개월)**
- 전체 시스템 아키텍처 설계
- 데이터베이스 ERD 완성
- UI/UX 디자인 전체 완료

**3단계 - 구현 (3개월)**
- 모든 기능을 동시에 개발
- 백엔드, 프론트엔드 팀이 각각 작업

**4단계 - 테스트 & 배포 (1개월)**
- 전체 시스템 통합 테스트
- 한 번에 전체 배포

**결과**: 안정적이지만 중간에 "모바일 최적화 필요"라는 요구사항이 생겨 추가 비용과 일정 지연 발생

---

### 2. 애자일 개발 적용 사례

**스타트업 B2C 쇼핑몰 (3개월 MVP 개발)**

**Sprint 1 (2주): 핵심 기능**
- 상품 목록 조회
- 간단한 주문 기능
- 매주 고객 피드백 수집

**Sprint 2 (2주): 개선**
- 사용자가 "장바구니 기능 필요"라고 피드백
- 즉시 장바구니 기능 추가
- 결제 방식 간소화

**Sprint 3 (2주): 확장**
- 리뷰 시스템 추가
- 모바일 반응형 개선

**Sprint 4-6: 지속적 개선**
- 실 사용자 데이터 분석
- A/B 테스트를 통한 UI 최적화
- 매 스프린트마다 새로운 기능 추가/수정

**결과**: 빠른 시장 진입, 사용자 피드백 반영으로 높은 만족도 달성

---

### 3. 도메인 주도 설계 적용 사례

**대형 이커머스 플랫폼 리뉴얼**

**도메인 분석 단계**
```
전문가 인터뷰: "주문이란 무엇인가?"
- 마케팅팀: "프로모션과 할인이 적용되는 과정"
- 물류팀: "배송지와 상품을 연결하는 과정"  
- 결제팀: "금액 정산과 결제 처리 과정"
```

**유비쿼터스 언어 정의**
- "주문" → "Order"
- "장바구니에 담기" → "AddToCart"
- "결제 대기" → "PendingPayment"
- "배송 준비" → "PreparingShipment"

**도메인 모델 설계**
```java
// 비즈니스 규칙이 코드에 명확히 표현됨
public class Order {
    public void apply(Discount discount) {
        if (!this.isEligibleForDiscount(discount)) {
            throw new InvalidDiscountException("이 주문은 할인 대상이 아닙니다");
        }
        // 할인 적용 로직
    }
}
```

**바운디드 컨텍스트 분리**
- 주문 도메인: 주문 생성, 상태 관리
- 결제 도메인: 결제 처리, 환불
- 배송 도메인: 배송지 관리, 배송 추적
- 상품 도메인: 재고 관리, 상품 정보

**결과**: 복잡한 비즈니스 로직을 체계적으로 관리, 각 팀이 독립적으로 개발 가능

## 실무에서의 선택 기준

**폭포수가 적합한 경우:**
- 정부 프로젝트, 금융권 시스템
- 요구사항이 명확하고 변경이 적은 경우
- 규제가 엄격한 환경

**애자일이 적합한 경우:**
- 스타트업, 새로운 서비스 개발
- 사용자 피드백이 중요한 B2C 서비스
- 빠른 시장 출시가 필요한 경우

**DDD가 적합한 경우:**
- 복잡한 비즈니스 로직을 가진 시스템
- 여러 팀이 협업하는 대규모 프로젝트
- 도메인 전문가와의 긴밀한 협업이 가능한 환경

**현실적으로는** 대부분의 프로젝트에서 이 방법들을 조합해서 사용합니다. 예를 들어, "애자일 방식으로 개발하되 복잡한 비즈니스 로직은 DDD 원칙을 적용"하는 하이브리드 접근법을 많이 활용합니다.
