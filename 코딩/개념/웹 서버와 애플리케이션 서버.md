웹 서버는 **정적 콘텐츠**를, 애플리케이션 서버는 **동적 콘텐츠**를 처리하는 것이 가장 큰 차이입니다.

**레스토랑**에 비유하면 아주 쉽게 이해할 수 있습니다. 🍽️

- **웹 서버 (Web Server) = 홀 직원**: 손님에게 메뉴판을 주거나 물을 가져다주는 것처럼, **미리 준비된 것(정적 콘텐츠)**을 요청에 따라 그대로 전달합니다.
    
- **애플리케이션 서버 (Application Server, WAS) = 주방**: 손님의 주문(요청)을 받아 재료(데이터)를 손질하고, 레시피(비즈니스 로직)에 따라 요리하여 **새로운 음식(동적 콘텐츠)**을 만들어 냅니다.
    

---

### ## 웹 서버 (Web Server)

**핵심 역할**은 HTML 파일, CSS 파일, 이미지처럼 **이미 만들어져 있는 정적 파일**을 HTTP 프로토콜을 통해 클라이언트(웹 브라우저)에게 전달하는 것입니다.

- **처리 내용**: 정적 콘텐츠 (HTML, CSS, JavaScript, 이미지, 문서 파일 등)
    
- **주요 기능**: 클라이언트의 요청을 받아 해당 파일을 찾아 전달, 간단한 접근 제어
    
- **예시**: **Apache HTTP Server**, **Nginx**
    

---

### ## 애플리케이션 서버 (Application Server, WAS)

**핵심 역할**은 클라이언트의 요청에 따라 **서버 내부에서 비즈니스 로직을 실행**하고, 그 결과를 **동적으로 생성**하여 클라이언트에게 전달하는 것입니다. 이 과정에서 데이터베이스 조회나 다른 시스템과의 연동이 일어납니다.

- **처리 내용**: 동적 콘텐츠 (JSP, Servlet, Python, Node.js 등으로 실행된 결과)
    
- **주요 기능**: 비즈니스 로직 실행, 데이터베이스 연동, 트랜잭션 관리 등
    
- **포함 관계**: 일반적으로 웹 서버의 기능을 포함하고 있거나, 웹 서버와 연동하여 사용됩니다.
    
- **예시**: **Apache Tomcat**, **JBoss/WildFly**, **Node.js**, **Gunicorn (Python)**
    

---

### ## 둘은 어떻게 함께 사용되나요?

현대의 웹 시스템은 대부분 두 서버를 함께 사용합니다. 사용자의 모든 요청을 웹 서버가 먼저 받은 후, 정적인 요청은 스스로 처리하고 동적인 요청만 애플리케이션 서버에 전달하는 **리버스 프록시(Reverse Proxy)** 구조를 사용합니다.

`사용자 ↔ 웹 서버 (Nginx) ↔ 애플리케이션 서버 (Tomcat) ↔ 데이터베이스`

이렇게 구성하면 웹 서버가 동적 요청을 기다리는 동안 다른 정적 요청들을 빠르게 처리할 수 있어 **성능과 안정성**이 크게 향상됩니다.

| 구분         | 웹 서버 (Web Server)         | 애플리케이션 서버 (Application Server) |
| ---------- | ------------------------- | ------------------------------ |
| **주요 목적**  | 정적 콘텐츠 제공                 | 비즈니스 로직 실행 및 동적 콘텐츠 생성         |
| **처리 콘텐츠** | HTML, CSS, 이미지 등          | JSP, Servlet, DB 조회 결과 등       |
| **대표 예시**  | Apache HTTP Server, Nginx | Apache Tomcat, Node.js, JBoss  |
